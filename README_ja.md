# zmk-input-processor-scroll-inertia

ZMKのトラックボール用、iOS風の慣性スクロールモジュール。

> **加速はもうやめよう。滑らせよう。**
>
> ZMKでトラックボールのスクロールを快適にしようとすると、多くの人は
> *加速度* を足します(ボールを速く回す → スクロール量が増える)。
> このモジュールは違うアプローチを取ります。足すのは *慣性* です
> (ボールを離す → スクロールが続いて、ゆっくり消えていく)。
>
> 両方を組み合わせて使うこともできますが、まだ「慣性だけ」を試した
> ことがないなら、まずそれを試してみてください。
> これは単なる機能ではなく、今までのトラックボール体験に足りなかった
> ピースです。

---

## このモジュールがやること

ZMKの入力プロセッサのひとつです。スクロールイベントを一軸分監視し、
「フリックして離した」タイミングを検知して、そこから滑らかに減衰する
スクロール出力を続けます。

減衰のしくみは2ステップでシンプル：まず速度に「1よりちょっと
小さい数字」を掛けて、それから「ごく小さい数字」を引きます。
前者がiOS風のなめらかな減衰、後者が「小さなフリックがいつまでも
滑らない」ためのひと押しです。全体で数字3つだけ。
(必要なら、高速と低速で別の減衰率を使う多段階モードもあります。)

## このモジュールがやらないこと

- *スクロールの加速* はしません。入力を増幅する処理は含みません。
- 「軸自動判定」や「スマートな軸ロック」の実装ではありません。
  理由は次のセクションを読んでください。

## なぜレイヤーを複数に分けるのか

### トラックボールの真実

トラックボールは、連続的な2Dの回転を返します。回転そのものには
「縦軸」も「横軸」もなく、どんな回し方にも必ずXとYの両方が混ざって
います。ユーザーが縦にスクロールしたいのか、横にスクロールしたい
のかは、**本人の頭の中にしか情報が無く、回転データには含まれて
いません**。

### なぜ自動判定ではうまくいかないのか

単一レイヤーで判定する実装は、この「意図」を回転データから
推測しようとします。大抵は動きますが、以下の状況で必ず破綻します:

- 45度付近の角度(どっちの軸が優勢か決められない)
- 方向反転時(前の方向のピークが判定を狂わせる)
- ゆっくりの動き(ノイズと本物の区別ができない)
- アクティブスクロールから慣性への引き継ぎ(状態を正確に持ち越す必要がある)

これらは「頑張って直せるバグ」ではなく、**そもそも入力に存在しない
情報を読み取ろうとしていること**から来る構造的な限界です。

### 解決策: 推測しない、宣言してもらう

このモジュールは推測をきっぱりやめて、ユーザーに**レイヤーキーで
意図を宣言してもらう**設計にしました:

- 縦スクロール用レイヤー → `axis = <1>`
- 横スクロール用レイヤー → `axis = <2>`
- 必要なら自由2Dスクロール用 → `axis = <0>`

これで軸自動判定にまつわる問題は全部消えます。うまく扱えたから
ではなく、**そもそも起こり得ない構造**になるからです。

### *「でも、そんなに余ったレイヤーが無いんだけど...」*

大丈夫です。一番綺麗なのは専用レイヤーを作る方法ですが、スクロール
レイヤー1つだけでも100%確実な軸ロックは実現できます。次の2つの
修飾キーを組み合わせます:

- `swap-mod` (例: `MOD_LSFT`) — 押している間だけ出力軸を反転。
  縦スクロールレイヤー + Shiftで横スクロール。
- `unlock-mod` (例: `MOD_RGUI`) — 押している間だけ軸ロックを解除
  して両軸フリー(自由2Dスクロール)。

どちらも「レイヤー切り替え」ではなく「修飾キー」で意図を宣言する、
というだけの違いです。根本の原則は変わりません: 推測しない、軸が
詰まらない、どんな角度でも挙動が一意に決まる。

> **修飾キーは慎重に選んでください。**押している間は修飾キーが
> ホスト(PC/Mac)に送られるので、ホストのショートカットと干渉する
> ことがあります(ほとんどのアプリで Cmd/Ctrl + スクロール = ズーム
> です)。安全策は2つ:
>
> 1. 普段のスクロール系ショートカットで使われにくい修飾キーを選ぶ。
>    右側の修飾キー、特に `MOD_RGUI` は比較的安全です。
> 2. もっと安全なのは、**その修飾キーをスクロールレイヤー内だけに
>    割り当てる**方法です。他のレイヤーでは透過(`&trans`)にしておけ
>    ば、修飾キーはスクロール操作中以外は絶対に送られません。
>    普段のタイピングを邪魔しません。

## インストール方法

### `west.yml` に追加

`config/west.yml` の `projects:` リストに以下を追加:

```yaml
manifest:
  remotes:
    - name: mjmjm0101
      url-base: https://github.com/mjmjm0101
  projects:
    - name: zmk-input-processor-scroll-inertia
      remote: mjmjm0101
      revision: main
```

### `*.conf` ファイル

何も追加する必要はありません。オーバーレイに対応するDTノードが
あれば、モジュールは自動で有効になります。

### `*.overlay` ファイル

**推奨の配置**: `&zip_scroll_scaler` の**手前**に置き、
`scale` と `scale-div` をそのscalerの2つの引数と同じ値にします。

```dts
#include <dt-bindings/zmk/modifiers.h>

/ {
    scroll_inertia_v: scroll_inertia_v {
        compatible = "zmk,input-processor-scroll-inertia";
        #input-processor-cells = <0>;

        /* axis: 0 = 両軸(自由2D),
                  1 = Y軸のみ(縦スクロール),
                  2 = X軸のみ(横スクロール) */
        axis      = <1>;
        layer     = <4>;          /* 縦スクロール用のレイヤー番号 */

        /* 後段の zip_scroll_scaler の引数と合わせる */
        scale     = <4>;
        scale-div = <675>;

        /* オプション: Shiftを押している間だけ出力軸を反転 */
        swap-mod  = <(MOD_LSFT|MOD_RSFT)>;

        /* オプション: 右GUIを押している間だけ両軸フリー (2Dスクロール) */
        unlock-mod = <MOD_RGUI>;
    };

    /* 専用の自由2Dレイヤーは任意 — 上の unlock-mod を使えば、縦
     * スクロールレイヤー内にいながら2Dスクロールに切り替えられる
     * ので、両方を用意する必要はない。お好みで選択してください。 */
    scroll_inertia_free: scroll_inertia_free {
        compatible = "zmk,input-processor-scroll-inertia";
        #input-processor-cells = <0>;
        axis      = <0>;          /* 両軸 — 斜めスクロール可能 */
        layer     = <5>;          /* 自由2Dスクロール用レイヤー (キャンバス操作など) */
        scale     = <4>;
        scale-div = <675>;
    };

    trackball_listener: trackball_listener {
        compatible = "zmk,input-listener";
        device = <&trackball>;
        input-processors = <&pointer_accel /* など */>;

        scroller_v {
            layers = <4>;
            input-processors = <
                &zip_y_scaler (-1) 1
                &zip_xy_to_scroll_mapper
                &scroll_inertia_v
                &zip_scroll_scaler 4 675
            >;
        };

        scroller_free {
            layers = <5>;
            input-processors = <
                &zip_y_scaler (-1) 1
                &zip_xy_to_scroll_mapper
                &scroll_inertia_free
                &zip_scroll_scaler 4 675
            >;
        };
    };
};
```

> **`pointer_accel` についての注意:** iOS風のスクロールは、ボール
> を回している間は「物理的な動きと画面のスクロールが 1:1 で一致する」
> ことが前提です。スクロールチェーンに加速度プロセッサが入っていると、
> ゆっくりした動きが大きな値に増幅されて、フリックしていないのに
> 慣性が誤発動します。カーソル用の(デフォルトの)チェーンには
> `pointer_accel` を残してもかまいませんが、**スクロール用のチェーンからは
> 外してください**。

## プロパティ一覧

値について一言: いくつかのプロパティは「1000分のいくつ」(千分率)の
値で指定します。例えば減衰率 `990` は「毎tickごとに速度に0.990を掛ける」
という意味です。キーボード側で浮動小数点演算を避けるための書き方です。

**強制される安全規則** (モジュールが内部で書き換えて自分自身を守る値):

- `scale-div = 0` は `1` 扱い (そのままだとゼロ除算)
- `tick = 0` は `1` ms 扱い (そのままだと暴走)

**推奨範囲** (利用者の責任。モジュールは自動補正しません。範囲外の値を
入れれば範囲外の挙動になります):

- `gain` + `blend` は `1000` になるように。合計が1000を超えると
  EMAが発散、下回ると入力に追いつかなくなります。
- `decay-fast` / `decay-slow` / `decay-tail` は `[800, 999]` の範囲。
  1000以上にすると速度が減らず慣性が止まりません。
- `slow` ≤ `fast` を維持。逆転させると中速域がスキップされます。

| プロパティ | デフォルト | 説明 |
|---|---|---|
| `gain` | `300` | 新しいイベントがスムーズ化された速度にどれだけ影響するか(1000分率)。`blend` と組み合わせて移動平均を作ります: `velocity = (event * gain + velocity * blend) / 1000` |
| `blend` | `700` | 前回の速度をどれだけ残すか(1000分率)。`gain + blend` は 1000 になるようにします。 |
| `start` | `40` | 慣性が発動するのに必要な「フリックの勢い」の最低値。縦横合わせた速さを見るので、斜めでも真っ直ぐでも**同じ強さで弾けば同じように発動**します。軽めの操作で意図せず発動するようなら上げてください。 |
| `move` | `80` | 慣性が発動するのに必要な「フリックの長さ」の最低値。ほんの少しの操作で誤発動しないようにするためのフィルタです。軽いスクロールでも慣性が起きるなら上げ、ちゃんとフリックしても起きないなら下げます。 |
| `release` | `24` | 何ミリ秒イベントが来なかったら「指を離した」と見なすか。125Hzだと3フレーム相当の24msが目安。 |
| `fast` | `0` | 高速域と中速域の境界。`0`にすると段階分けを無効化(= 単一カーブ)。 |
| `decay-fast` | `990` | 高速域の減衰率(1000分率/tick)。990なら `velocity *= 0.990` を毎tick実行。 |
| `decay-slow` | `990` | 中速域の減衰率。 |
| `slow` | `0` | 中速域と低速(tail)域の境界。`0`にすると段階分けを無効化。 |
| `decay-tail` | `990` | 低速(tail)域の減衰率。 |
| `friction` | `35` | 慣性に「ほんのり効く一定ブレーキ」を加える値。毎tick一定量ずつ速度を引くので、速度が遅いほど相対的に強く効きます(= 小さいフリックが早めに止まる)。大きいフリックの高速域はほぼ影響を受けません。値の単位は scroll unit の permille (1/1000) で、`friction=1000` なら毎tick 1単位を引く計算。`0`で完全に無効(純指数減衰)。詳しくは下の注意書きを参照。 |
| `stop` | `7` | 速度がこれを下回ると慣性は停止。下の注意書きも参照。 |
| `scale` | `1000` | 出力スケールの分子。 |
| `scale-div` | `1000` | 出力スケールの分母。毎tick `velocity × scale / scale-div` を内部accumulatorに足し、整数スクロール単位が溜まるたびに送信します。 |
| `limit` | `600` | 速度の安全上限(raw)。強すぎるフリックで暴走するのを防ぎます。 |
| `span` | `6000` | 慣性の最大持続時間(ms)。これは暴走防止の安全装置で、自然な減衰なら大抵もっと早く終わります。 |
| `tick` | `8` | 慣性ループの間隔(ms)。センサーのポーリングレートに合わせます(8ms ≈ 125Hz)。 |
| `axis` | `0` | `0` = 両軸(斜め可)、`1` = Y軸のみ(縦)、`2` = X軸のみ(横)。 |
| `layer` | `-1` | `0`以上にすると、そのレイヤーがオフになった瞬間に慣性も止まります。`-1`でレイヤーチェック無効。 |
| `swap-mod` | `0` | 修飾キーのビットマスク(`MOD_LSFT`, `MOD_RCTL` など)。押している間、出力軸が反転(V↔H)します。`0`で無効。 |
| `unlock-mod` | `0` | 修飾キーのビットマスク。押している間、軸ロックが解除されて両軸のスクロールが通ります(= `axis=0` 相当)。`swap-mod` より優先。`0`で無効。 |

### `stop` についての注意

慣性の**終わり際**の質感は、`stop` 単体ではなく、`stop` と `scale`、
`scale-div` の組み合わせで決まります:

```
終了時の出力頻度(Hz) ≒ stop × scale × 125 / scale-div
```

大体5Hzを下回ると、整数単位のスクロールが「カチッ、カチッ」と
離散的に感じられてしまいます。このしきい値より上に保つように
チューニングします。

| `scale/scale-div` | 推奨 `stop`(約5Hzで終了) |
|---|---|
| `1:1` (1000/1000) | `1` |
| `1:10` (100/1000) | `10` |
| `4:675` (推奨配置) | `7` |
| `1:100` (10/1000) | `100` |

直感に反しますが、`stop` を下げるとtailは**悪化**します(良くは
なりません) — 速度がさらに低い領域まで到達して、スクロール単位と
単位の間がどんどん空いてしまうからです。`stop` は下げるのではなく
**上げる**のが正解で、動きが滑らかなうちに綺麗にカットオフできます。

### `friction` についての注意

`decay-*` は「速度 × 減衰率」の掛け算なので、大きく弾いても小さく
弾いても**尻尾の長さの比率は同じ**になります。そのため「大きく
弾いた時はちょうどいいのに、小さく弾いた時だけ慣性が残りすぎる」
という違和感が出やすいです。

`friction` はそこに一定量の引き算を足します。常に同じ量ずつ引くので、
速度が遅いほど影響が強く見えます:

| 速度の場面 | 指数減速 (`decay-fast=990`) | `friction=35` (デフォルト) | 支配的 |
|---|---|---|---|
| 大弾きのピーク (vel=600) | −6/tick (1.0%) | −0.035/tick (0.006%) | 指数減速 |
| 中速 (vel=100) | −1/tick (1.0%) | −0.035/tick (0.035%) | 指数減速 |
| 尻尾 (vel=20) | −0.2/tick (1.0%) | −0.035/tick (0.175%) | 指数減速 |
| 停止直前 (vel=5) | −0.05/tick (1.0%) | −0.035/tick (0.7%) | ほぼ互角 |

小さい弾きは低速域で過ごす時間の割合が多いので、friction の効きが
相対的に大きく、結果として早めに止まります。

**注意点:** friction は慣性の間じゅう引き続けるので、あまり大きい値に
すると大きいフリックの終わりも短くなってしまいます。デフォルトの
`35` は「小弾きだけ短くして大弾きは触らない」バランスで選んだ値です。

**精度:** `4` 以下の値は内部で0に丸められます。完全に無効にしたい
場合は `0` を指定してください(小さい正の値ではなく)。

**調整の方向:** 小弾きをもっとキビキビ止めたい → `100` くらいまで
上げる。長いテイルが欲しい → `0` まで下げる。`~500` を超えると
大弾きにも目に見える短縮が出ます。

### `scale` / `scale-div` についての注意

このプロセッサの慣性出力は、チェーン内の後段にあるscalerを**通り
ません**(HIDに直接書き込む実装のため)。アクティブスクロールと
慣性スクロールのスピードを揃えるために、後段のscalerの引数と同じ
値を `scale` と `scale-div` に入れてください:

```dts
&scroll_inertia_v
&zip_scroll_scaler 4 675     /* ← ここの2引数を */

scroll_inertia_v {
    scale     = <4>;          /*    ここに合わせる */
    scale-div = <675>;
};
```

デフォルトの `1000` / `1000` (1:1) が正しいのは、このプロセッサを
チェーンの**末尾**(scalerの後ろ)に置いた場合だけです。それは推奨
配置ではありません — scalerの手前に置いた方が、速度トラッカーが
大きい値を扱えて精度が良く、scalerが内部accumulatorを貯めている
間に出す `value=0` のイベントに邪魔されません。

## チューニングのコツ

実戦で得た「知っているとラクになる」ポイントをいくつか。

### 全体のスクロール量を減らす

慣性が無い状態では、「スクロールが遅い」のを補うために
`zip_scroll_scaler` を派手な比率にしたり、`&msc` を3-5回まとめて
送るマクロを使っていた人が多いと思います。慣性があると、一回の
フリックで既に長距離が進むので、そこに派手なスケールを足すと
スクロールが速すぎて制御不能に感じます。

目安: 慣性を導入するタイミングで、それまでのスクロール量を半分
程度に下げる。`zip_scroll_scaler` の引数 **と** このプロセッサの
`scale` / `scale-div` の両方を更新してください(この2つは常に一致
させます)。

### スクロールチェーンから `pointer_accel` を外す

iOS風のスクロールは「ボールの動きと画面のスクロールが 1:1 で一致する」
ことが前提です。`pointer_accel`(やそれに類する加速度プロセッサ)が
スクロールチェーンに入っていると、ゆっくりの動きが大きな値に増幅
されてフリックっぽく見え、意図しない慣性の発動につながります。
カーソル用チェーンは残してかまいませんが、スクロール用チェーン
からは外してください。

### 自分のトラックボールに合わせて `start` と `move` を調整

この2つは「どのくらいの勢いで慣性が起きるか」を決めます。正しい
値はトラックボールのCPI、加速度設定、そしてあなたのフリックの
強さで変わります。

- 最初は `start = 40`, `move = 80` (デフォルト — 1000 CPI PMW3610で
  ベクトル magnitude アーム向けに再調整した値)から始めます。
- 軽めのスクロールでも慣性が起きてしまうなら: 両方を1.5-2倍に
  引き上げます。
- ちゃんとフリックしてるのに起きないなら: 30%くらい下げます。

2つのしきい値は独立に働きます — `start` は**ベクトル速度**(両軸合成)
のフィルタ、`move` は累積距離のフィルタ。両方を満たさないと慣性は
起きません。`start` は magnitude で判定するので、雑な角度で弾いても
軸に揃ったフリックと同じ物理強度で閾値に達します。

### 好みに合わせて減衰率を選ぶ

デフォルトの `decay-* = 990` はiOS風の、長く滑らかなtailを作ります。

| 目指す感触 | 設定 |
|---|---|
| iOS風の長い滑走感 | `990-995` (単一カーブ、3つとも同じ値) |
| キビキビした短めの慣性 | `980-985` |
| 2段階(最初は速く、tailは穏やか) | `decay-fast = 985`, `decay-slow = decay-tail = 993`, `fast = 60` |

まずは単一カーブから始めるのをおすすめします。大抵はそれで90%
満足できて、必要なときだけ多段階に手を出せば十分です。

### `friction` で小さいフリックを早めに止める

デフォルトの `friction = 35` で既に小さなフリックに軽いブレーキが
かかっています。それでも感触が合わない場合は、上げたり下げたりで
調整できます。参考値(フリックの大小別にどのくらい早く止まるか):

| 値 | 大きめフリック (v=600) | 小さめフリック (v=75) | 備考 |
|---|---|---|---|
| `0` | 3.5秒 | 1.9秒 | 純指数。大小の比率は同じまま。 |
| `35` (デフォルト) | 約3.3秒 (−6%) | 約1.7秒 (−12%) | 小フリックだけテイルが短くなる。大フリックはほぼそのまま。 |
| `100` | 2.8秒 (−20%) | 1.3秒 (−30%) | もう一段キビキビ。デフォルトで物足りない時に。 |
| `200` | 2.5秒 (−30%) | 1.0秒 (−47%) | 強め。大フリックにも効き始める。 |
| `500`以上 | 1.9秒以下 | 1秒以下 | iOS風の感触には強すぎ。大フリックが引きずられる印象に。 |

小フリックがまだ長い → `25〜50` 刻みで上げる。大フリックが重く
感じる → 下げる、究極的には `0` で指数減衰だけに。`4` 以下は内部で
0に丸められるので、`1〜3` を指定する意味はありません。

### tailを伸ばしすぎない

スクロール単位は整数(1単位刻み)で、「0.5単位スクロール」のような
中間値はHIDでは送れません。秒間5単位くらいを下回ると、スクロール
単位同士の間隔が視覚的にカクつきとして感じられてしまいます —
内部の速度カーブがどれだけ滑らかでも、です。上の**`stop` についての
注意**にある式と推奨値の表を参照してください。

直しかたは `stop` を**上げる**こと、下げることではありません。
速度をゼロまで到達させようとするのが一番質感が悪くなります —
間延びしたカチッ、カチッ、の後に無音が続くことになります。

### `tick` をセンサーのポーリングレートに合わせる

デフォルトの `tick = 8` (≈ 125Hz) は PMW3610 に合わせた値です。60Hzの
センサーなら `tick = 16`、それ以上のレートならもっと小さくします。
ズレていても動作に支障は出ませんが、合わせておくとアクティブ
スクロールから慣性への引き継ぎがより自然になります。

## 動作の仕組み

このモジュールは各インスタンスを3状態のステートマシンとして実装
しています：

```
        ┌─── ジェスチャータイムアウト / stop_detect失敗 ───┐
        │                                                  ▼
        │                                                IDLE
        │                                                  │ 最初の tracked イベント
        │                                                  ▼
        │       ┌── 反転 / クロス軸ブレーク ──┐
        │       │                              │
        │       ▼                              │
    TRACKING ──(発動条件+減速検知)──────────▶ COASTING
        ▲                                         │
        │                                         │ vel < stop,
        └────────── (IDLE経由) ──────────────────┘ span超過,
                                                    layer無効
```

### 各状態の役割

1. **IDLE（待機）。** 何も起きていない状態。最初のスクロール
   イベントが来ると TRACKING に移ります。
2. **TRACKING（追跡）。** ユーザーがボールを転がしている最中。
   毎イベントごとに「いまどのくらいの速さか(EMA)」と「これまでの
   最大速度(peak)」を更新しながら、慣性を発動すべきかを判断します。
   判定は縦と横の**速度を合成した値**(ベクトル強度、ほぼ
   `√(x²+y²)`)を使うので、雑な角度で弾いても軸に揃ったフリックと
   同じ強さで発動します。合成速度のピークが `start` 以上、移動量が
   `move` 以上、イベントが10個以上集まると「発動可能」モードに入り、
   その後に合成速度がピークの85%を3回連続で下回った時点で COASTING
   に切り替わります。指を離すなどで急に止まった場合は、`release`
   ms 後のバックアップ判定(stop_detect)が同じ条件で拾います。
3. **COASTING（慣性で滑走中）。** 切り替わった瞬間、慣性の初速を
   合成速度のピークに揃えます。これで斜めフリックも真っ直ぐな
   フリックも**同じ物理強度で弾けば同じだけ滑ります**。`tick` ms
   ごとにタイマーが発火し、速度を `decay` で減らして `friction` を
   引き、たまったらスクロールとして PC に送信します。途中で同方向の
   イベントが来れば吸収(必要に応じて速度を合わせる)、逆方向なら
   TRACKING に戻して直接操作を受け付けます。速度が `stop` 未満に
   なる / レイヤーがオフになる / `span` に達する のいずれかで
   IDLE に戻ります。

### 自動リセット

以下が起きるとどの状態からでも IDLE に戻ります：

- **ジェスチャータイムアウト** — 100ms イベントが来なかったら、
  次のイベントは「新しいジェスチャーの始まり」として扱う
- **慣性の取り残し検出** — COASTING 中にイベントが2tick分以上
  来なかったら、もう物理操作から切り離されていると判断して終了
- **クロス軸ブレーク** — COASTING 中に、使っていない方の軸の
  入力が `move` を超えたら、ユーザーがわざと別方向を操作して
  いる → 直接操作を返す(single-axis モードのみ)

### 安全装置

COASTING 中の同方向入力吸収が連続50回(125Hz で約400ms)続いたら、
ユーザーが慣性越しに能動的に回し続けていると判断して TRACKING に
戻します。通常はクロス軸ブレークの方が先に効くので、これは
「どうしても状態が詰まった時の最後の保険」という位置づけです。
