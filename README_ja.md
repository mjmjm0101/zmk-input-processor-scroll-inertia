# zmk-input-processor-scroll-inertia

[![test](https://github.com/mjmjm0101/zmk-input-processor-scroll-inertia/actions/workflows/test.yml/badge.svg)](https://github.com/mjmjm0101/zmk-input-processor-scroll-inertia/actions/workflows/test.yml)

ZMKのトラックボール用、iOS風の慣性スクロールモジュール。

> **加速じゃなくて滑らせよう。**

![scroll demo](https://github.com/user-attachments/assets/fc8f89e9-8212-4a2c-80cf-3898f67d8bdf)

>
> ZMKでトラックボールのスクロールを快適にしようとすると、多くの人は
> *加速度* を足します（ボールを速く回す → スクロール量が増える）。
> このモジュールは違うアプローチを取ります。足すのは *慣性* です
> （ボールを離す → スクロールが続いて、ゆっくり消えていく）。
>
> 両方を組み合わせて使うこともできますが、まだ「慣性だけ」を試した
> ことがないなら、まずそれを試してみてください。
> これは単なる機能ではなく、今までのトラックボール体験に足りなかった
> ピースです。

*※ v0.5.x 以前より慣性のかかり方を滑らかにしました。*

---

## このモジュールがやること

ZMKの入力プロセッサのひとつです。
スクロールイベントを一軸分監視し、「フリックして離した」タイミングを検知して、そこから滑らかに減衰するスクロール出力を続けます。

減衰のしくみは2ステップでシンプルです。
まず速度に「1よりちょっと小さい数字」を掛けて、それから「ごく小さい数字」を引きます。
前者がiOS風のなめらかな減衰、後者が「小さなフリックがいつまでも滑らない」ためのひと押しです。
必要な数字は3つだけです。
（必要なら、高速と低速で別の減衰率を使う多段階モードもあります。）

## このモジュールがやらないこと

- *スクロールの加速* はしません。
  入力を増幅する処理は含みません。
- 「軸自動判定」や「スマートな軸ロック」の実装ではありません。
  理由は次のセクションを読んでください。

## なぜレイヤーを複数に分けるのか

### トラックボールの真実

トラックボールは、連続的な2Dの回転を返します。
回転そのものには「縦軸」も「横軸」もなく、どんな回し方にも必ずXとYの両方が混ざっています。
ユーザーが縦にスクロールしたいのか、横にスクロールしたいのかは、**本人の頭の中にしか情報がなく、回転データには含まれていません**。

### なぜ自動判定ではうまくいかないのか

単一レイヤーで判定する実装は、この「意図」を回転データから推測しようとします。
大抵は動きますが、以下の状況で必ず破綻します。

- 45度付近の角度（どっちの軸が優勢か決められない）
- 方向反転時（前の方向のピークが判定を狂わせる）
- ゆっくりの動き（ノイズと本物の区別ができない）
- アクティブスクロールから慣性への引き継ぎ（状態を正確に持ち越す必要がある）

これらは「頑張って直せるバグ」ではなく、**そもそも入力に存在しない情報を読み取ろうとしていること**から来る構造的な限界です。

### 解決策：推測しない、宣言してもらう

このモジュールは推測をきっぱりやめて、ユーザーに**レイヤーキーで意図を宣言してもらう**設計にしました。

- 縦スクロール用レイヤー → `axis = <1>`
- 横スクロール用レイヤー → `axis = <2>`
- 必要なら自由2Dスクロール用 → `axis = <0>`

これで軸自動判定にまつわる問題は全部消えます。
うまく扱えたからではなく、**そもそも起こり得ない構造**になるからです。

### *「でも、そんなに余ったレイヤーが無いんだけど...」*

大丈夫です。
一番綺麗なのは専用レイヤーを作る方法ですが、スクロールレイヤー1つだけでも100%確実な軸ロックは実現できます。
次の2つの修飾キーを組み合わせます。

- `swap-mod` （例: `MOD_LSFT`） — 押している間だけ出力軸を反転。
縦スクロールレイヤー + Shiftで横スクロール。
- `unlock-mod` （例: `MOD_RGUI`） — 押している間だけ軸ロックを解除
して両軸フリー（自由2Dスクロール）。

どちらも「レイヤー切り替え」ではなく「修飾キー」で意図を宣言する、というだけの違いです。
根本の原則は変わりません。
 推測しない、軸が詰まらない、どんな角度でも挙動が一意に決まる。

> **修飾キーは慎重に選んでください。**押している間は修飾キーが
> ホスト（PC/Mac）に送られるので、ホストのショートカットと干渉する
> ことがあります（ほとんどのアプリで Cmd/Ctrl + スクロール = ズーム
> です）。安全策は2つあります。
>
> 1. 普段のスクロール系ショートカットで使われにくい修飾キーを選ぶ。
>    右側の修飾キー、特に `MOD_RGUI` は比較的安全です。
> 2. もっと安全なのは、**その修飾キーをスクロールレイヤー内だけに
>    割り当てる**方法です。他のレイヤーでは透過（`&trans`）にしておけ
>    ば、修飾キーはスクロール操作中以外は絶対に送られません。
>    普段のタイピングを邪魔しません。

## インストール方法

### `west.yml` に追加

`config/west.yml` の `projects:` リストに以下を追加します。

```yaml
manifest:
  remotes:
    - name: mjmjm0101
      url-base: https://github.com/mjmjm0101
  projects:
    - name: zmk-input-processor-scroll-inertia
      remote: mjmjm0101
      revision: main
```

### ZMK のバージョン

**ZMK v0.3.0**（`zmkfirmware/zmk` タグ `v0.3`）で動作確認しています。
`config/west.yml` の `zmk` プロジェクトを該当リビジョン（または互換バージョン）に固定してください：

```yaml
projects:
  - name: zmk
    remote: zmkfirmware
    revision: v0.3
    import: app/west.yml
```

### `*.conf` ファイル

必須の追加はありません。
オーバーレイに対応するDTノードがあれば、モジュールは自動で有効になります。
`CONFIG_ZMK_POINTING` は本モジュールの Kconfig から自動で `select` されます（マウス HID API を叩くため必須）。

覚えておくと良い設定が2つあります：

- **`CONFIG_ZMK_POINTING`** — 必須ですが自動で有効化されます。ユーザーが書く必要はありません。
- **`CONFIG_ZMK_POINTING_SMOOTH_SCROLLING=y`** — 任意。
  HID Resolution Multiplier が有効化され、対応ホスト（比較的新しい macOS / Linux / Windows）側でスクロール出力を高解像度として解釈してくれるため、慣性の末尾がよりなめらかに感じられます。
  有効化するなら、ホスト側の単位解釈が変わるのでオーバーレイの `scale` / `scale-div` と `stop` を見直したほうがいいです。
  ホストが対応しているなら推奨、していないなら付けなくても問題ありません。

#### 分割キーボードの場合

分割ビルドでは、このプロセッサは **central 側に置く必要があります**。
ZMK のマウス HID API は central ロールでしか compile されず、peripheral 側にプロセッサを置いても HID 出力は central からしか出ないので意味がありません。

DT ノードが peripheral ビルドに含まれると、モジュールはコンパイル時に明示的なエラーで停止します（silent skip はしません）。
プロセッサの DT ノード（および参照している input-listener）を central 専用 overlay に置くか、共有 overlay を side-scope するかして解決してください。

### `*.overlay` ファイル

**推奨の配置：** `&zip_scroll_scaler` の**手前**に置き、
`scale` と `scale-div` をそのscalerの2つの引数と同じ値にします。

```dts
#include <dt-bindings/zmk/modifiers.h>

/ {
    scroll_inertia_v: scroll_inertia_v {
        compatible = "zmk,input-processor-scroll-inertia";
        #input-processor-cells = <0>;

        /* axis: 0 = 両軸（自由2D）,
                  1 = Y軸のみ（縦スクロール）,
                  2 = X軸のみ（横スクロール） */
        axis      = <1>;
        layer     = <4>;          /* 縦スクロール用のレイヤー番号 */

        /* 後段の zip_scroll_scaler の引数と合わせる */
        scale     = <4>;
        scale-div = <675>;

        /* オプション: Shiftを押している間だけ出力軸を反転 */
        swap-mod  = <(MOD_LSFT|MOD_RSFT)>;

        /* オプション: 右GUIを押している間だけ両軸フリー （2Dスクロール） */
        unlock-mod = <MOD_RGUI>;
    };

    /* 専用の自由2Dレイヤーは任意 — 上の unlock-mod を使えば、縦
     * スクロールレイヤー内にいながら2Dスクロールに切り替えられる
     * ので、両方を用意する必要はない。お好みで選択してください。 */
    scroll_inertia_free: scroll_inertia_free {
        compatible = "zmk,input-processor-scroll-inertia";
        #input-processor-cells = <0>;
        axis      = <0>;          /* 両軸 — 斜めスクロール可能 */
        layer     = <5>;          /* 自由2Dスクロール用レイヤー （キャンバス操作など） */
        scale     = <4>;
        scale-div = <675>;
    };

    trackball_listener: trackball_listener {
        compatible = "zmk,input-listener";
        device = <&trackball>;
        input-processors = <&pointer_accel /* など */>;

        scroller_v {
            layers = <4>;
            input-processors = <
                &zip_y_scaler (-1) 1
                &zip_xy_to_scroll_mapper
                &scroll_inertia_v
                &zip_scroll_scaler 4 675
            >;
        };

        scroller_free {
            layers = <5>;
            input-processors = <
                &zip_y_scaler (-1) 1
                &zip_xy_to_scroll_mapper
                &scroll_inertia_free
                &zip_scroll_scaler 4 675
            >;
        };
    };
};
```

> **`pointer_accel` についての注意:** iOS風のスクロールは、ボール
> を回している間は「物理的な動きと画面のスクロールが 1:1 で一致する」
> ことが前提です。スクロールチェーンに加速度プロセッサが入っていると、
> ゆっくりした動きが大きな値に増幅されて、フリックしていないのに
> 慣性が誤発動します。カーソル用の（デフォルトの）チェーンには
> `pointer_accel` を残してもかまいませんが、**スクロール用のチェーンからは
> 外してください**。

## プロパティ一覧

値について一言： いくつかのプロパティは「1000分のいくつ」（千分率）の値で指定します。
例えば減衰率 `990` は「毎tickごとに速度に0.990を掛ける」という意味です。
キーボード側で浮動小数点演算を避けるための書き方です。

**強制される安全規則** （モジュールが内部で書き換えて自分自身を守る値）:

- `scale-div = 0` は `1` 扱い （そのままだとゼロ除算）
- `tick = 0` は `1` ms 扱い （そのままだと暴走）

**推奨範囲** （利用者の責任。モジュールは自動補正しません。範囲外の値を
入れれば範囲外の挙動になります）:

- `gain` + `blend` は `1000` になるように。合計が1000を超えると
  EMAが発散、下回ると入力に追いつかなくなります。
- `decay-fast` / `decay-slow` / `decay-tail` は `[800, 999]` の範囲。
  1000以上にすると速度が減らず慣性が止まりません。
- `slow` ≤ `fast` を維持。逆転させると中速域がスキップされます。

| プロパティ | デフォルト | 説明 |
|---|---|---|
| `gain` | `300` | 新しいイベントがスムーズ化された速度にどれだけ影響するか（1000分率）。`blend` と組み合わせて移動平均を作ります: `velocity = (event * gain + velocity * blend) / 1000` |
| `blend` | `700` | 前回の速度をどれだけ残すか（1000分率）。`gain + blend` は 1000 になるようにします。 |
| `start` | `40` | 慣性が発動するのに必要な「フリックの勢い」の最低値。縦横合わせた速さを見るので、斜めでも真っ直ぐでも**同じ強さで弾けば同じように発動**します。軽めの操作で意図せず発動するようなら上げてください。 |
| `move` | `80` | 慣性が発動するのに必要な「フリックの長さ」の最低値。ほんの少しの操作で誤発動しないようにするためのフィルタです。軽いスクロールでも慣性が起きるなら上げ、ちゃんとフリックしても起きないなら下げます。 |
| `release` | `24` | 何ミリ秒イベントが来なかったら「指を離した」と見なすか。125Hzだと3フレーム相当の24msが目安。 |
| `fast` | `0` | 高速域と中速域の境界。`0`にすると段階分けを無効化（= 単一カーブ）。 |
| `decay-fast` | `990` | 高速域の減衰率（1000分率/tick）。990なら `velocity *= 0.990` を毎tick実行。 |
| `decay-slow` | `990` | 中速域の減衰率。 |
| `slow` | `0` | 中速域と低速（tail）域の境界。`0`にすると段階分けを無効化。 |
| `decay-tail` | `990` | 低速（tail）域の減衰率。 |
| `friction` | `35` | 慣性に「ほんのり効く一定ブレーキ」を加える値。毎tick一定量ずつ速度を引くので、速度が遅いほど相対的に強く効きます（= 小さいフリックが早めに止まる）。大きいフリックの高速域はほぼ影響を受けません。値の単位は scroll unit の permille （1/1000） で、`friction=1000` なら毎tick 1単位を引く計算。`0`で完全に無効（純指数減衰）。詳しくは下の注意書きを参照。 |
| `stop` | `7` | 速度がこれを下回ると慣性は停止。下の注意書きも参照。 |
| `scale` | `1000` | 出力スケールの分子。 |
| `scale-div` | `1000` | 出力スケールの分母。毎tick `velocity × scale / scale-div` を内部accumulatorに足し、整数スクロール単位が溜まるたびに送信します。 |
| `limit` | `600` | 速度の安全上限（raw）。強すぎるフリックで暴走するのを防ぎます。 |
| `span` | `6000` | 慣性持続時間（ms）の暴走防止キャップで、自然な減衰なら大抵もっと早く終わります。同方向で強度が `start` 以上のイベントが来ると新しいフリック扱いでこのタイマーが再計測されるため、強めの同方向連打がこのキャップで途切れることはありません。 |
| `tick` | `8` | 慣性ループの間隔（ms）。センサーのポーリングレートに合わせます（8ms ≈ 125Hz）。 |
| `axis` | `0` | `0` = 両軸（斜め可）、`1` = Y軸のみ（縦）、`2` = X軸のみ（横）。 |
| `layer` | `-1` | `0`以上にすると、そのレイヤーがオフになった瞬間に慣性も止まります。`-1`でレイヤーチェック無効。 |
| `swap-mod` | `0` | 修飾キーのビットマスク（`MOD_LSFT`, `MOD_RCTL` など）。押している間、出力軸が反転（V↔H）します。`0`で無効。 |
| `unlock-mod` | `0` | 修飾キーのビットマスク。押している間、軸ロックが解除されて両軸のスクロールが通ります（= `axis=0` 相当）。`swap-mod` より優先。`0`で無効。 |

### 上級者向けチューニング

内部ステートマシンのタイミングを調整するつまみです。
デフォルトは1000 CPI PMW3610 @ 125Hz で調整済みで、一般的なセットアップはほぼこれで問題ありません。
自前のハードで具体的な問題を特定した場合だけ触ってください。
範囲外の値は自動補正されません。

| プロパティ | デフォルト | 推奨範囲 | 説明 |
|---|---|---|---|
| `min-events` | `10` | `[3, 30]` | 慣性発動を許可する前に必要なトラック軸イベントの最小数。EMAが収束する時間を確保し、一時的なスパイクを弾きます。低すぎるとノイズで発動し、高すぎると短い意図的フリックで発動しません。直前のCOASTINGから直接TRACKINGに戻ってきた場合は、このチェックは自動的にスキップされるので、高速反転フリックがこの門番で止まることはありません。 |
| `decel-samples` | `3` | `[1, 10]` | TRACKING → COASTING の判定で、ピーク以下が連続で必要な回数。`1`にすると単発の速度低下で即発動、大きくすると遷移が遅れます。 |
| `decel-ratio` | `850` | `[700, 950]` | 減速判定のピーク比（千分率）。`850` = 現在の速度がピークの85%を下回れば減速と判定。大きいと微ブレで誤発動、小さいと穏やかな release で発動が遅れます。 |
| `peak-decay` | `990` | `[950, 999]` | ピーク速度のdecay率（千分率/イベント）。速度がピークを下回っている間、ピークをゆっくり下方へドリフトさせて、初期過渡の異常ピーク残留を防ぎます。**1000以上は危険** — ピークが発散します。 |
| `gesture-timeout` | `100` | `[50, 500]` | この時間イベントが来なかったら、次のイベントは新ジェスチャー扱い（全トラッキング状態リセット）。前セッションの速度/ピークが新セッションに持ち越されるのを防ぎます。 |
| `suppress-limit` | `50` | `[20, 200]` | COASTING中に同方向イベントを連続で吸収した回数の上限。超えるとTRACKINGに戻ります。`50` ≈ 400ms @ 125Hz。低すぎると有効な長いコーストで脱落、高すぎると詰まった状態が長引きます。 |
| `handoff-ms` | `100` | `[50, 300]` | 慣性開始後、同方向で慣性速度以下のイベントを**無条件に**吸収する期間（ms）。フリック直後のボール自然減速が慣性を妨げないようにするための窓です。窓を過ぎた後は、同方向イベントが**慣性の半分未満**の時だけ下流へ通します — ボール自然減速は慣性より減衰が遅く、その尾は慣性速度の近傍に留まるため吸収側に残り、通常のフリック挙動は変わりません。一方で明らかに弱い新ユーザー入力（慣性中に軽く指を添えて回した場合など）は半分を大きく下回るので通過し、末尾のサブユニット出力と重なって画面が止まって見える症状を防ぎます。慣性速度を超えるイベントは窓の内外を問わず従来通り bump します。 |

### `stop` についての注意

慣性の**終わり際**の質感は、`stop` 単体ではなく、`stop` と `scale`、
`scale-div` の組み合わせで決まります。

```
終了時の出力頻度（Hz） ≒ stop × scale × 125 / scale-div
```

大体5Hzを下回ると、整数単位のスクロールが「カチッ、カチッ」と離散的に感じられてしまいます。
このしきい値より上に保つようにチューニングします。

| `scale/scale-div` | 推奨 `stop`（約5Hzで終了） |
|---|---|
| `1:1` （1000/1000） | `1` |
| `1:10` （100/1000） | `10` |
| `4:675` （推奨配置） | `7` |
| `1:100` （10/1000） | `100` |

直感に反しますが、`stop` を下げるとtailは**悪化**します（良くはなりません） — 速度がさらに低い領域まで到達して、スクロール単位と単位の間がどんどん空いてしまうからです。
`stop` は下げるのではなく**上げる**のが正解で、動きが滑らかなうちに綺麗にカットオフできます。

### `friction` についての注意

`decay-*` は「速度 × 減衰率」の掛け算なので、大きく弾いても小さく弾いても**尻尾の長さの比率は同じ**になります。
そのため「大きく弾いた時はちょうどいいのに、小さく弾いた時だけ慣性が残りすぎる」という違和感が出やすいです。

`friction` はそこに一定量の引き算を足します。
常に同じ量ずつ引くので、速度が遅いほど影響が強く見えます。

| 速度の場面 | 指数減速 （`decay-fast=990`） | `friction=35` （デフォルト） | 支配的 |
|---|---|---|---|
| 大弾きのピーク （vel=600） | −6/tick （1.0%） | −0.035/tick （0.006%） | 指数減速 |
| 中速 （vel=100） | −1/tick （1.0%） | −0.035/tick （0.035%） | 指数減速 |
| 尻尾 （vel=20） | −0.2/tick （1.0%） | −0.035/tick （0.175%） | 指数減速 |
| 停止直前 （vel=5） | −0.05/tick （1.0%） | −0.035/tick （0.7%） | ほぼ互角 |

小さい弾きは低速域で過ごす時間の割合が多いので、friction の効きが相対的に大きく、結果として早めに止まります。

**注意点:** friction は慣性の間ずっと引き続けるので、あまり大きい値にすると大きいフリックの終わりも短くなってしまいます。
デフォルトの`35` は「小弾きだけ短くして大弾きは触らない」バランスで選んだ値です。

**精度:** `4` 以下の値は内部で0に丸められます。
完全に無効にしたい場合は `0` を指定してください（小さい正の値ではなく）。

**調整の方向:** 小弾きをもっとキビキビ止めたい → `100` くらいまで上げる。
長いテイルが欲しい → `0` まで下げる。
`~500` を超えると大弾きにも目に見える短縮が出ます。

### `scale` / `scale-div` についての注意

このプロセッサの慣性出力は、チェーン内の後段にあるscalerを**通りません**（HIDに直接書き込む実装のため）。
アクティブスクロールと慣性スクロールのスピードを揃えるために、後段のscalerの引数と同じ値を `scale` と `scale-div` に入れてください:

```dts
&scroll_inertia_v
&zip_scroll_scaler 4 675     /* ← ここの2引数を */

scroll_inertia_v {
    scale     = <4>;          /*    ここに合わせる */
    scale-div = <675>;
};
```

デフォルトの `1000` / `1000` （1:1） が正しいのは、このプロセッサをチェーンの**末尾**（scalerの後ろ）に置いた場合だけです。
それは推奨配置ではありません — scalerの手前に置いた方が、速度トラッカーが大きい値を扱えて精度が良く、scalerが内部accumulatorを貯めている間に出す `value=0` のイベントに邪魔されません。

## チューニングのコツ

実戦で得た「知っているとラクになる」ポイントをいくつか。

### 全体のスクロール量を減らす

慣性が無い状態では、「スクロールが遅い」のを補うために`zip_scroll_scaler` を派手な比率にしたり、`&msc` を3-5回まとめて送るマクロを使っていた人が多いと思います。
慣性があると、一回のフリックで既に長距離が進むので、そこに派手なスケールを足すとスクロールが速すぎて制御不能に感じます。

目安：慣性を導入するタイミングで、それまでのスクロール量を半分程度に下げる。
`zip_scroll_scaler` の引数 **と** このプロセッサの`scale` / `scale-div` の両方を更新してください（この2つは常に一致させます）。

### スクロールチェーンから `pointer_accel` を外す

iOS風のスクロールは「ボールの動きと画面のスクロールが 1:1 で一致する」ことが前提です。
`pointer_accel`（やそれに類する加速度プロセッサ）がスクロールチェーンに入っていると、ゆっくりの動きが大きな値に増幅されてフリックっぽく見え、意図しない慣性の発動につながります。
カーソル用チェーンは残してかまいませんが、スクロール用チェーンからは外してください。

### 自分のトラックボールに合わせて `start` と `move` を調整

この2つは「どのくらいの勢いで慣性が起きるか」を決めます。
正しい値はトラックボールのCPI、加速度設定、そしてあなたのフリックの強さで変わります。

- 最初は `start = 40`, `move = 80` （デフォルト — 1000 CPI PMW3610で
  ベクトル magnitude アーム向けに再調整した値）から始めます。
- 軽めのスクロールでも慣性が起きてしまうなら、両方を1.5〜2倍に
  引き上げます。
- ちゃんとフリックしてるのに起きないなら、30%くらい下げます。

2つのしきい値は独立に働きます — `start` は**ベクトル速度**（両軸合成）のフィルタ、`move` は累積距離のフィルタ。
両方を満たさないと慣性は起きません。
`start` は magnitude で判定するので、雑な角度で弾いても軸に揃ったフリックと同じ物理強度で閾値に達します。

### 好みに合わせて減衰率を選ぶ

デフォルトの `decay-* = 990` はiOS風の、長く滑らかなtailを作ります。

| 目指す感触 | 設定 |
|---|---|
| iOS風の長い滑走感 | `990-995` （単一カーブ、3つとも同じ値） |
| キビキビした短めの慣性 | `980-985` |
| 2段階（最初は速く、tailは穏やか） | `decay-fast = 985`, `decay-slow = decay-tail = 993`, `fast = 60` |

まずは単一カーブから始めるのをおすすめします。
大抵はそれで90%満足できて、必要なときだけ多段階に手を出せば十分です。

### `friction` で小さいフリックを早めに止める

デフォルトの `friction = 35` で既に小さなフリックに軽いブレーキがかかっています。
それでも感触が合わない場合は、上げたり下げたりで調整できます。
参考値（フリックの大小別にどのくらい早く止まるか）:

| 値 | 大きめフリック （v=600） | 小さめフリック （v=75） | 備考 |
|---|---|---|---|
| `0` | 3.5秒 | 1.9秒 | 純指数。大小の比率は同じまま。 |
| `35` （デフォルト） | 約3.3秒 （−6%） | 約1.7秒 （−12%） | 小フリックだけテイルが短くなる。大フリックはほぼそのまま。 |
| `100` | 2.8秒 （−20%） | 1.3秒 （−30%） | もう一段キビキビ。デフォルトで物足りない時に。 |
| `200` | 2.5秒 （−30%） | 1.0秒 （−47%） | 強め。大フリックにも効き始める。 |
| `500`以上 | 1.9秒以下 | 1秒以下 | iOS風の感触には強すぎ。大フリックが引きずられる印象に。 |

小フリックがまだ長い → `25〜50` 刻みで上げる。
大フリックが重く感じる → 下げる、究極的には `0` で指数減衰だけに。
`4` 以下は内部で0に丸められるので、`1〜3` を指定する意味はありません。

### tailを伸ばしすぎない

スクロール単位は整数（1単位刻み）で、「0.5単位スクロール」のような中間値はHIDでは送れません。
秒間5単位くらいを下回ると、スクロール単位同士の間隔が視覚的にカクつきとして感じられてしまいます —
内部の速度カーブがどれだけ滑らかでも、です。
上の**`stop` についての注意**にある式と推奨値の表を参照してください。

直し方は `stop` を**上げる**こと、下げることではありません。
速度をゼロまで到達させようとするのが一番質感が悪くなります —
間延びしたカチッ、カチッ、の後に無音が続くことになります。

### `tick` をセンサーのポーリングレートに合わせる

デフォルトの `tick = 8` (≈ 125Hz) は PMW3610 に合わせた値です。
60Hzのセンサーなら `tick = 16`、それ以上のレートならもっと小さくします。
ズレていても動作に支障は出ませんが、合わせておくとアクティブスクロールから慣性への引き継ぎがより自然になります。

## 動作の仕組み

このモジュールは各インスタンスを3状態のステートマシンとして実装しています：

```
    IDLE  ──▶  TRACKING  ──▶  COASTING  ──▶  IDLE
```

各遷移の詳細：

| 遷移 | きっかけ |
|---|---|
| `IDLE → TRACKING` | 最初のスクロールイベント |
| `TRACKING → COASTING` | ピークのベクトル強度 ≥ `start` かつ累積移動量 ≥ `move` かつイベント `min-events` 回以上（直前のCOASTINGから戻ってきた場合はこの条件をスキップ）→ そのうえで減速が `decel-samples` 回連続で検出された時点。急停止の場合は `release` ms 後に発火する stop_detect が同じ条件で判定 |
| `COASTING → TRACKING` | **single-axis**: 逆方向イベント、クロス軸ブレーク（COASTING 中の非トラック軸の累積入力が `move` 超過）、または同方向吸収が `suppress-limit` 回連続（保険）。**AXIS_BOTH**: 部分リセットで両軸とも無効になった時点（片軸の逆方向はその軸だけ部分リセット — COASTING の項を参照）、または同方向吸収が `suppress-limit` 回連続。 |
| `COASTING → IDLE` | 速度が `stop` 未満になる / `span` 到達 / レイヤー無効 |
| *任意の状態* `→ IDLE` | ジェスチャータイムアウト（`gesture-timeout` ms イベントなし）、または慣性の取り残し検出（COASTING 中に2tick分イベントなし） |

### 各状態の役割

1. **IDLE（待機）。** 何も起きていない状態。
   最初のスクロールイベントが来ると TRACKING に移ります。
2. **TRACKING（追跡）。** ユーザーがボールを転がしている最中。
   毎イベントごとに「いまどのくらいの速さか（EMA）」と「これまでの最大速度（peak）」を更新しながら、慣性を発動すべきかを判断します。
   判定は縦と横の**速度を合成した値**（ベクトル強度、ほぼ`√(x²+y²)`）を使うので、雑な角度で弾いても軸に揃ったフリックと同じ強さで発動します。
   合成速度のピークが `start` 以上、移動量が`move` 以上、イベントが `min-events` 個以上集まると「発動可能」モードに入り、その後に合成速度がピークの `decel-ratio`/1000 を`decel-samples` 回連続で下回った時点で COASTING に切り替わります。
   指を離すなどで急に止まった場合は、`release` ms 後のバックアップ判定（stop_detect）が同じ条件で拾います。
   **post-coast 緩和：**
   直前のCOASTINGから直接TRACKINGに戻ってきた場合（逆方向フリック、クロス軸ブレーク、suppress-limit 安全装置、慣性の取り残し検出のいずれか）、次の arm では `min-events` のチェックをスキップします。
   直前にちゃんと慣性を起こせた＝意図的なジェスチャだった証拠があるので、短い高速反転フリックで毎回10イベント積み直す必要がなくなります。
   `start` と `move` はそのまま効いているので、閾値以下の揺れで誤発動することはありません。
3. **COASTING（慣性で滑走中）。** 切り替わった瞬間、慣性の初速を**現在の合成速度**（EMA 値）に揃えます。
   これで斜めフリックも真っ直ぐなフリックも**同じ物理強度で弾けば同じだけ滑ります**。
   ピークではなく「現在の」値を使うので、リリース直前に見えていたスクロール速度から連続して慣性に入り、発動時に加速したように見えることはありません。
   `tick` ms ごとにタイマーが発火し、速度を`decay` で減らして（`move` ギリのジェスチャーほど高速域の減衰が速まるよう軽く補正されます）`friction` を引き、たまったらスクロールとして PC に送信します。
   途中で同方向のイベントが来れば吸収（必要に応じて速度を合わせる）、逆方向なら TRACKING に戻して直接操作を受け付けます。
   速度が `stop` 未満になる / レイヤーがオフになる / `span` に達する のいずれかで IDLE に戻ります。

   ここまでの説明は single-axis モード（`axis = <1>` または `<2>`）の挙動です。
   `axis = <0>`（AXIS_BOTH）は 2Dパン向けの専用 COASTING ハンドラを使うので、次項で違いを説明します。

#### AXIS_BOTH（2Dパン）固有の COASTING 挙動

`axis = <0>` はマップ・デザインツールなどの 2Dパン用途を想定しており、COASTING の挙動が single-axis モードといくつか異なります：

- **軸別の coast 初速。**
  突入時、`vel_x` と `vel_y` はトラッキングの EMA 値をそのまま引き継ぎます（片軸への畳み込みなし）。斜め flick は斜めに滑走します。
  ベクトル長が `limit` を超える場合のみ両軸を比例縮小して物理速度の上限を `limit` に揃えます（single-axis と同じキャップ）。
- **magnitude ベースの decay。**
  decay と friction はベクトル長に対して適用され、軸は比例スケールされます。45° の coast は軸方向の coast と同じ物理減速率になります。
  zone 境界（`fast`, `slow`）と stop gate もベクトル長で判定。
- **fresh axis の pass-through。**
  `vel == 0` の軸にイベントが来た場合（片軸だけの coast 中）、逆方向ではなく「新しいパン方向」として扱い、そのまま通します。他軸の coast は継続。
- **逆方向時の部分リセット。**
  片軸が逆方向になった場合、その軸の vel / peak / accum だけゼロ化し、直交軸の coast は継続します。
  両軸とも無効になった場合のみ TRACKING に戻ります（あるいは `suppress-limit` 到達時）。
- **クロス軸ブレークなし。**
  両軸ともトラック対象なので、single-axis の「非トラック軸が能動的に回された時の保険」は不要です。

同方向の bump / absorb / weak-passthrough は各軸で single-axis と同じ扱いです（共通ヘルパ）。

### 自動リセット

以下が起きるとどの状態からでも IDLE に戻ります：

- **ジェスチャータイムアウト** — `gesture-timeout` ms イベントが
  来なかったら、次のイベントは「新しいジェスチャーの始まり」として扱う
- **慣性の取り残し検出** — COASTING 中にイベントが2tick分以上
  来なかったら、もう物理操作から切り離されていると判断して終了
- **クロス軸ブレーク** — COASTING 中に、使っていない方の軸の
  入力が `move` を超えたら、ユーザーがわざと別方向を操作して
  いる → 直接操作を返す（single-axis モードのみ）

### 安全装置

COASTING 中の同方向入力吸収が連続 `suppress-limit` 回（デフォルト`50` = 125Hz で約400ms）続いたら、ユーザーが慣性越しに能動的に回し続けていると判断して TRACKING に戻します。
通常はクロス軸ブレークの方が先に効くので、これは「どうしても状態が詰まった時の最後の保険」という位置づけです。
